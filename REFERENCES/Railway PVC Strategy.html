A vertical contract-calculation system to replace manual Excel workflows and obsolete the IRPVC legacy SaaS. Four hard requirements: deterministic calculation traceability, spreadsheet-grade user control, contract-specific configurable rules, and immutable audit snapshots.

Market gap: Neither Excel nor IRPVC solves the upstream problem. The contractual truth is distributed across tender tabulations, MBs, running bills, recovery sheets, and index tables. The winning product normalizes this chain — PVC is a downstream output, not the entry point.

Calculation Logic — W Derivation Formula Detail
W = OnAccountBillAmount − CementAmount (Cement!H19 → item × consumption_factor × rate) − SteelAnglesAmount (Steel!H11 → angle/channel/beam bucket) − SteelPlatesAmount (Steel!H12 → plate bucket) − SteelOtherAmount (Steel!H14 → other sections) − TechnicalWithheld (if applicable under contract) − ExtraItemAmount (if item marked non-eligible) PVC_component = W × component_weight × (Q_avg_index − base_index) / base_index Where Q_avg_index = (index_month1 + index_month2 + index_month3) / 3 for the quarter containing the bill's measurement date. Steel sub-formula (per bucket): PVC_steel = bucket_amount × [ 0.10 × (labour_Δ/base) + 0.50 × (steel_commodity_Δ/base) + 0.10 × (plant_Δ/base) + 0.10 × (fuel_Δ/base) + 0.05 × (materials_Δ/base) ] (Residual 0.15 = non-variable, no PVC applied) Carry-forward proration: paid_ratio = paid_qty_bill_n / recorded_qty_total first_bill_bucket = bucket_total × paid_ratio carry_forward = bucket_total × (1 − paid_ratio) → B/F into bill n+1

Edge Cases & Blind Spots

Critical operational, technical, and regulatory failure modes omitted or underspecified in both source reports. Each carries an architectural mitigation.

Critical Offline Operational Constraints
Both reports assume persistent internet connectivity. Railway project sites (tunnels, remote civil works, hill sections) have zero reliable connectivity. Billing engineers run calculations on-site, not in offices. If the SaaS requires a live connection for every calculation, it cannot replace Excel in practice — Excel wins by default for field users.
Offline-capable Progressive Web App with local IndexedDB cache for the current contract's rule snapshot, bill data, and index table. Calculation engine runs client-side in a Web Worker (pure JS port of the Python engine) with sync-on-reconnect. Conflict resolution: server is authoritative; offline runs are flagged as pending-sync until reconciled.
Critical Retroactive Index Revision (RBI/JPC Revised Publications)
RBI and JPC regularly revise previously published indices — sometimes months after original publication. IRPVC's index store does not surface revision status. If an approved PVC run used a superseded index value, the calculation is wrong — but no alert fires. This is a regulatory audit risk that can trigger claim disputes with Railways.
IndexObservation entity carries a revision_flag and revised_at timestamp. Platform scans all approved PVC runs nightly against current index store. If a run used an index value that has since been revised, it surfaces a "stale index" alert on the contract dashboard and notifies the contract admin. Re-calculation requires explicit acknowledgment and superseding run creation.
Critical Quarter Boundary Ambiguity — Bill Date vs. Execution Period
The source reports do not resolve a fundamental domain question: is the PVC quarter determined by the bill's measurement date, the bill submission date, or the actual execution period of the work? For bills prepared after contract completion covering earlier execution periods, the correct quarter may be 2-3 quarters earlier than the current date. IRPVC appears to use measurement date, but this is unconfirmed and may vary by Railway zone.
Quarter determination is a first-class configurable rule in the PVC RuleSet — three modes: (a) measurement date, (b) bill date, (c) override by operator. Platform warns if bill date and execution period span multiple quarters, prompting explicit quarter selection. Approved quarter is stored as a snapshot field — not derivable post-hoc. Domain interviews with 2+ railway zones required before shipping this feature as locked.
High Missing Month Index — Bill Submitted Before All Quarter Months Published
If a running bill is prepared in the first month of a quarter, indices for the other two months of that quarter are not yet published. IRPVC's behavior in this case is unknown. Using partial averages silently is a calculation error. Blocking the run entirely creates operational paralysis.
Platform supports provisional calculation mode: a PVC run with missing indices is flagged as PROVISIONAL — it can be exported for contractor reference but cannot receive final approval. When the missing index is published and entered, the system auto-prompts a superseding final run. Provisional runs are watermarked in all exports.
High Negative PVC (De-escalation) and Offset Accounting
Neither report addresses what happens when the current index is lower than the base month index — producing negative PVC, meaning the contractor owes money back to Railways. This is a recovery event, not a payment. The treatment of negative PVC offsetting against positive components within the same bill is also unresolved.
PVC component results carry signed values. Aggregation respects component-level signs. A bill where net PVC is negative is flagged as a "de-escalation bill" with explicit alert. Platform produces both contractor-facing and railway-facing output formats. Recovery accounting for de-escalation must be confirmed against GCC clause 46A interpretation — configurable as recoverable-from-next-bill vs. immediate-offset.
High Railway Zone Variance in PVC Templates
GCC clause 46A is a national standard but Railway zones (Western, Central, Southern, etc.) maintain zone-specific component weight tables, classification hierarchies, and index publication sources. IRPVC's handling of multi-zone portfolios is unclear. A contractor operating across zones needs zone-specific rule templates — a single global template is wrong.
PVC RuleSet is zone-tagged. Platform ships with a curated template library covering confirmed zone variants. Contract setup requires zone selection, which auto-loads the correct rule template as a starting point. Template library is versioned and maintained as a platform data asset — this is a long-term competitive moat.
High Audit Trail Compliance for Submission Defense
Railway field accounts require that a submitted PVC bill can be defended at audit — meaning every number must trace to a source document, a rule, and an index publication. Current Excel workflow has zero provenance. IRPVC produces a formatted PDF but likely cannot answer "which index publication issue was this value from?"
Every cell in a PVC run output carries a provenance object: {source: 'IndexObservation#42', rule: 'PVCRuleSet#7', bill_line: 'BillLine#183', operator: 'user_id', timestamp}. Audit export is a separate artifact — a structured JSON or linked-PDF document where every figure hyperlinks to its source. This is the product's primary defense against dispute.
High GST Treatment and PVC Base Interaction
Running bills show GST as a separate line. PVC should be calculated on pre-GST amounts. However, if the bill is GST-inclusive and the operator uses the gross figure as the PVC base, the result is inflated. Neither report explicitly models GST-exclusive normalization as a pipeline step.
Contract setup captures GST mode (inclusive/exclusive). Bill ingestion pipeline normalizes all amounts to pre-GST basis before PVC base derivation. GST component is stripped and tracked separately. PVC run output includes a GST-basis-used field in the provenance record.
High Contract Time Extension Impact on PVC Eligibility (17A/17B)
Extensions under clauses 17A and 17B can alter PVC eligibility — specifically, whether PVC is admissible for work executed in the extension period beyond the original completion date. IRPVC surfaces extensions as a data entity but the rule coupling between extension type and PVC eligibility is not fully specified.
Extension entity carries pvc_eligibility_in_extension flag with three states: full / capped / excluded. When a bill's execution period overlaps an extension window, the system applies the corresponding rule. Extension-period bills are visually distinguished in the dashboard and exports.
Critical Legacy Excel Import Safety
The lowest-resistance GTM path is importing existing contractor workbooks. However, Excel workbooks in this domain contain hidden formulas, merged cells, named ranges, circular references, and operator-specific sheet structures. Blind parsing produces silently wrong output — a failure mode worse than not importing at all, because the contractor trusts the number.
Excel import operates in "review mode only" — it never auto-confirms parsed values. Parser extracts candidate values and presents them side-by-side with the raw cell reference and formula for human confirmation. Anomaly detection flags: mismatched totals, blank index cells, W values that differ from sum of components, negative eligible amounts. Import produces a confidence score per field. Low-confidence fields are blocked from auto-confirmation.

Go-To-Market Strategy

Persona-specific value propositions, migration triggers, and conversion mechanics to force displacement of Excel and IRPVC.

User Personas & Value Propositions
Billing Engineer
Primary operator
Spends 4-8 hours per bill manually transcribing quantities, averaging indices, managing carry-forwards, and producing print-ready output. Sole keeper of formula knowledge — departure is an institutional risk.
Reduce per-bill time to under 45 minutes. Every formula visible and auditable. Hand off to reviewers without explaining the workbook. Exit without taking institutional knowledge with you.
Contractor Principal
Management / P&L owner
No real-time visibility into PVC receivables across the portfolio. Cannot tell if PVC claims are complete or if money is being left on the table. Bill status lives in one engineer's spreadsheet.
Live portfolio dashboard: PVC receivable by contract, anomaly flags for under-claimed amounts, submission status and Railway approval tracking. Convert invisible cash flow into managed receivables.
Bidding Analyst
Pre-award analysis
Cannot model projected PVC receipts during bid evaluation. PVC upside is typically ignored in financial modelling because the calculation is too complex to run quickly.
Scenario modelling: given tender data and index projections, calculate expected PVC uplift per year of contract. Converts PVC from an afterthought into a bidding variable.
Accounts Operator
Recoveries & deductions
Tracks security deposit, IT, labour cess, water charges in separate registers. Cannot quickly reconcile gross bill vs. net payable vs. PVC base. Recovery classification affects PVC eligibility but this link is opaque.
Unified recovery register with explicit PVC-eligibility tag per recovery type. Auto-reconcile gross → net → PVC base. Reduce reconciliation time from hours to minutes.
Contract Admin
Setup & compliance
Manually reads GCC clause 46A, translates into Excel formula configuration. One misread = wrong calculations across all bills. Extra-item eligibility decisions undocumented and often contested.
Template library for common zone × work type configurations. AI-assisted clause extraction from agreement PDF. Every eligibility decision timestamped and attributed — defensible at audit.
Reviewer / Approver
Internal quality gate
Cannot verify formula correctness without opening the spreadsheet and manually tracing cells. No structured review workflow. Approval is informal — email or verbal.
Structured review mode: see calculation side-by-side with source documents and index publications. One-click flag → comment → approve workflow. Signed approval logged as immutable record.
Conversion Strategy — From Excel & IRPVC to Platform
EXCEL DISPLACEMENT PLAYBOOK
Hook
Free Excel import. Upload your existing workbook → platform validates your formulas and flags any errors it finds. The first interaction proves value by finding a mistake.
Trigger 1 — Formula Risk
A billing engineer leaves or is transferred. The next person cannot understand the workbook. Platform offers institutional knowledge continuity — this is the "burning platform" moment.
Trigger 2 — Audit Event
Railway field accounts question a PVC claim. Contractor cannot produce source traceability. One disputed claim worth ₹2-5L converts the entire org to the platform.
Trigger 3 — Scale
Contractor wins 3+ simultaneous contracts. Managing separate workbooks per contract becomes operationally impossible. Portfolio view is the conversion point for growing contractors.
Lock-in Mechanic
Historical PVC run archive + anomaly history + rule templates tied to specific tender types. After 6 months of data, switching cost exceeds any competitive offer.
IRPVC DISPLACEMENT PLAYBOOK
Positioning
IRPVC is a PVC calculator. This platform is a billing operating system. IRPVC starts at bill amounts — this platform starts at the measurement book. The upstream advantage is non-replicable without a rebuild.
Wallet Friction Exploit
IRPVC gates calculation behind a Razorpay wallet top-up at the critical final step. Subscription model with unlimited runs removes the anxiety moment contractors hate most.
Classification Persistence
IRPVC requires re-entering sub-classification mappings on each bill. Platform persists contract-level classification templates and auto-applies on subsequent bills. Hour-long data entry becomes 5 minutes.
Audit Trail Wedge
IRPVC produces a 9-page PDF but cannot answer provenance questions. Market platform as "audit-defensible" — specifically targeting contractors who have faced Railway account disputes.
Migration Path
IRPVC data migration tool: import existing contracts from IRPVC export format. Zero re-entry friction for switchers. Offer 60-day free trial explicitly targeting IRPVC users.

Strategic conclusion: The correct product is not a better PVC calculator. It is the first system-of-record for Indian Railway contractor billing — where PVC calculation is a deterministic, auditable downstream output of a fully traceable document-linked pipeline. The moat is built on data depth, rule templates, and audit defensibility — assets that require 12-24 months to replicate and compound in value with every approved run and every defended audit claim.
